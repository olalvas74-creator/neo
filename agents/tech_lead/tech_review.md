## 2. Элементы, требующие кастомного JS

### Минимальный JS (не проблема для производительности)

- ✅ **Sticky header с тенью** — IntersectionObserver, ~10 строк, defer.
- ✅ **Рейтинг (лайк/дизлайк)** — AJAX fetch, ~30 строк. Cookie/localStorage для защиты от повторного голосования.
- ✅ **Кнопка «Копировать ссылку»** — `navigator.clipboard.writeText()`, ~5 строк.
- ✅ **Toggle оглавления (свернуть/развернуть)** — ~5 строк или чистый CSS через `<details>/<summary>`.

### Средний JS (требует внимания)

- ⚠️ **Прогресс-бар чтения** — scroll listener с throttle/requestAnimationFrame. Оценка: ~20 строк, но нужна оптимизация через `requestAnimationFrame` чтобы не влиять на FPS. **Решение:** passive scroll listener + rAF, inline в `<head>` для исключения layout shift.
- ⚠️ **Sticky sidebar с активным пунктом оглавления** (подсветка текущей секции при скролле) — IntersectionObserver на каждый H2, ~40 строк. **Решение:** один IntersectionObserver с `threshold: [0, 1]`, defer-загрузка скрипта.
- ⚠️ **Автодополнение поиска** — потребует AJAX-запросов к WP REST API + debounce ввода + dropdown UI. Оценка: ~100 строк или плагин (SearchWP Live Ajax Search). **Решение:** если сроки жмут — плагин SearchWP; если нужен контроль — кастомный скрипт с `fetch()` + `AbortController` + debounce 300ms. Загрузка скрипта только на странице поиска.
- ⚠️ **Category bar padding при скролле** (8px -> 4px) — scroll listener. **Решение:** объединить с listener'ом тени header'а, один обработчик на оба эффекта.

### Вердикт по JS

- ✅ Общий объем кастомного JS оценивается в ~3-5 КБ (minified, gzipped). Это в пределах нормы и не угрожает Core Web Vitals. Все скрипты загружаются с `defer`, критического JS нет.

- ✅ `<main>` — основной контент. Аналогично — на уровне дизайна не фиксируется, реализуется в верстке.

- ✅ `<section>` — секции категорий на главной, секции на странице статьи (рейтинг, шеринг, автор, «читай также»).
- ✅ `<aside>` — sidebar на странице статьи.
- ✅ `<footer>` — подвал описан.
- ✅ **Иерархия H1-H6:**
  - Главная: H1 (заголовок блога), H2 (названия категорий) — корректно.
  - Категория: H1 (название категории) — корректно.

## 5. Core Web Vitals

### LCP (Largest Contentful Paint) < 2.5s

- ✅ **Hero-баннер на главной — одно статичное изображение**, не карусель. Это правильное решение: LCP-элемент предсказуем.
- ✅ Изображения в WebP формате.
- ⚠️ **Hero 400px высоты с фоновым изображением** — потенциальный LCP-элемент. **Решение:** `<img>` с `fetchpriority="high"` вместо `background-image`, preload через `<link rel="preload" as="image">`. Указать `width` и `height` явно для исключения layout shift.
- ✅ Шрифты Comfortaa, Nunito, Inter — 3 семейства. **Внимание:** это на грани. **Решение:** `font-display: swap` + preload только для Comfortaa Bold и Nunito Regular (критические), остальные начертания подгружать lazy.

### FID / INP (Interaction to Next Paint) < 200ms

- ✅ JS минимальный (~3-5 КБ), defer-загрузка. Проблем не ожидается.
- ✅ Нет тяжелых JS-фреймворков.
- ✅ Шеринг через share URL (без SDK соцсетей) — отличное решение.

### CLS (Cumulative Layout Shift) < 0.1

- ✅ CTA-баннер имеет `min-height: 180px` — предотвращает layout shift.
- ✅ Изображения карточек указаны с фиксированным соотношением 4:3 (400x300) — `aspect-ratio: 4/3` в CSS.
- ⚠️ **Sticky header 112px** — при загрузке может вызвать CLS, если контент рендерится до применения sticky. **Решение:** задать `min-height: 112px` для header с первого рендера, либо `padding-top: 112px` для body.
- ⚠️ **Шрифты (FOUT)** — при `font-display: swap` возможен CLS от смены шрифта. **Решение:** preload ключевых woff2-файлов, использовать `size-adjust` в `@font-face` для fallback-шрифтов, максимально близких по метрикам.

### Общий вердикт по CWV

- ✅ Дизайн совместим с зеленой зоной CWV при условии выполнения рекомендаций выше.

## 6. Конфликты между дизайном и техническими ограничениями

### Конфликт 1: Три семейства шрифтов

- ⚠️ Дизайн использует **Comfortaa** (заголовки), **Nunito** (основной текст), **Inter** (метаданные, мелкий текст). Три семейства = 6-10 файлов woff2 = 150-300 КБ.
- **Решение:** допустимо, но нужна стратегия загрузки. Comfortaa и Nunito — preload. Inter — async. Либо рассмотреть замену Inter на Nunito для мелкого текста (визуально похожи), сократив до 2 семейств. **Рекомендация дизайнеру:** оценить возможность замены Inter -> Nunito.

### Конфликт 2: Автодополнение поиска

- ⚠️ Описан dropdown с категориями, max 6 пунктов. WordPress нативный поиск не поддерживает live search. Потребует:
  - WP REST API endpoint (есть из коробки: `/wp-json/wp/v2/posts?search=`)
  - Кастомный JS (~100 строк)
  - Или плагин SearchWP
- **Не критично**, но увеличивает scope. Можно вынести в P2 и запустить без автодополнения.

### Конфликт 3: Permalink-структура с подкатегориями

- ⚠️ URL `/blog/{категория}/{подкатегория}/{slug}/` — WordPress по умолчанию не поддерживает трехуровневую структуру через стандартные пермалинки. `%category%` подставляет parent-категорию.
- **Решение:** Permalink Manager Lite плагин или кастомные rewrite rules. Тестировать на конфликты с основным сайтом (блог в подпапке `/blog/`).

### Конфликт 4: Блог в подпапке /blog/

- ⚠️ Если основной сайт НЕ на WordPress — потребуется настройка reverse proxy (nginx) или отдельная установка WP в подпапку. Если основной сайт тоже на WP — используется custom post type или page с blog template.
- **Уточнить:** какая CMS/платформа у основного сайта neofamily.ru.

### Конфликт 5: Border-radius 20px на карточках + 32px на hero

- ✅ Не конфликт. Gutenberg поддерживает кастомные border-radius через `theme.json`. Визуально — современный тренд, технически — без проблем.
